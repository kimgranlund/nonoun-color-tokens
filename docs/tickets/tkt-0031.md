# TKT-0031 — Migrate the 18 open file tickets to GitHub Issues; retire the file backend for new intake

## Summary
Requested: "let's start using GIT as the backend for issues/tickets/bugs/etc." Ratified as
**ADR-017** (`docs/reference/references/decision-records.md`) and wired into `CLAUDE.md`'s Layout
section as the routing-table row scribe's `/bug-report`/`/feature` Phase 0 reads — both already
land in this same change, so new intake already routes to GitHub Issues. What's left is the
one-shot migration: `docs/tickets/` currently holds 30 files (TKT-0001..0030), 12 `done` (frozen as
historical archive, per ADR-017 — not migrated) and **18 open** (TKT-0004, TKT-0013 through
TKT-0030) that need a GitHub Issue each, plus the label taxonomy those issues rely on.

Reporter rulings from intake (both AskUserQuestion-confirmed):
1. **Migrate all 18 open tickets now** (not a going-forward-only split — no parallel open-ticket
   stores).
2. **Ratify with an ADR** (done, ADR-017) rather than a bare routing-line change.

## Acceptance
- Two new labels exist on `kimgranlund/ultimate-tokens`: `kind:bug`, `kind:feature` (the repo's
  stock `bug`/`enhancement` labels stay — these are additive, not replacements, since ADR-017's
  contract needs the exact `kind:`/`size:` vocabulary the file frontmatter used) plus
  `size:small`, `size:big`.
- Each of the 18 open files (TKT-0004, TKT-0013..TKT-0030) becomes one `gh issue create`, title =
  the file's `# TKT-XXXX — …` heading (id stripped or kept at the reporter's preference — decide
  once, apply to all 18 for consistency), body = the file's own `##` sections verbatim
  (Summary/Acceptance/Links/Scope-Open/Findings), labels = that file's `kind:` + `size:` values
  translated to `kind:*`/`size:*` labels.
- Each migrated file gets a one-line pointer at its top (`> Migrated to GitHub Issue #NN,
  2026-07-17 — see ADR-017.`) rather than being deleted — the file stays as the historical record
  of what was captured before the backend moved, consistent with the 12 `done` files staying put.
- `docs/tickets/tkt-00XX.md` internal cross-references between the 18 (e.g. TKT-0013's `Links`
  section naming TKT-0011/TKT-0012) still resolve for a reader — either left as TKT-#### prose
  (both are now closed/migrated, so the number remains a valid historical pointer) or annotated
  with the matching Issue number; pick one convention and apply it uniformly.
- `project-docs` skill (`.claude/skills/project-docs/SKILL.md`) updated to route open-work
  questions ("what's open", "what tickets exist") to `gh issue list` first, `docs/tickets/` second
  (archive/history only).
- A repo-wide check (grep) confirms no script or skill parses `docs/tickets/*.md` frontmatter
  (`status:`/`size:`/`kind:`) expecting it to reflect current open/closed state — if one is found,
  it's updated to read GitHub Issues (`gh issue list --json`) instead.
- `npm test` stays green (this touches no engine/UI code — a docs+process migration — but the
  standing gate still applies before calling it done).

## Scope / Open
- **In scope**: label creation, 18 issue creations, the 18 files' migration pointers,
  `project-docs` skill routing update, the frontmatter-consumer grep sweep.
- **Non-goal**: the 12 already-`done` files (TKT-0001, 0002, 0003, 0005, 0006, 0007, 0008, 0009,
  0010, 0011, 0012) do NOT get Issues — ADR-017 freezes them as the historical archive, closed
  work doesn't need a live Issue.
- **Non-goal**: no change to how PRs/CI work — this is the issue/ticket/bug-report backend only,
  shipping-changes' PR flow is untouched.
- **Open (mechanical, build time)**: issue title convention — keep the `TKT-XXXX — ` prefix for
  continuity with the file archive's numbering, or drop it since Issues have their own `#NN`
  numbering and a prefix would be redundant. Recommend keeping it once, for this migration batch
  only, so a reader cross-referencing old prose ("see TKT-0013") can still find the issue by title
  search.
- **Open (mechanical, build time)**: whether migrated-file pointers go in every file's frontmatter
  (`migrated-to: "#NN"`) in addition to the prose line, for a machine-readable join — recommended
  if any tooling ever wants to walk the archive→Issue mapping automatically, optional otherwise.
