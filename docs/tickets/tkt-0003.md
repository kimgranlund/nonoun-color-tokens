# TKT-0003 — Preset gallery strips are neutral-dominant and generic, not opinionated per concept

## Summary
Every Color Categories PRESET card's color strip (`presetTile()` in `src/ui/app.js`) uses a FIXED
width template (`SAMPLED_W = [36, 19, 19, 16, 6, 4]`) over the first 6 enabled palettes — and because
`neutral` is always first in that array, the widest (36%) band in EVERY strip is always the low-chroma
neutral swatch. The palette's real character colors (primary/secondary/accent) get squeezed into the
remaining 64%, split six ways, regardless of the preset's own authored hierarchy. Result: strips for
visually distinct concepts (a Japanese alpine range vs. a Canadian glacial lake vs. Patagonian granite
towers) all read as similar grey/tan-heavy bands. Make each strip's composition track the preset's own
authored dominant/supporting/accent weighting, so it's evocative of that specific concept.

## Acceptance
- A preset's strip derives its per-swatch WIDTH from that preset's own `story.groups[].pct` (the
  authored dominant/supporting/accent shares, e.g. 55/35/10) via each palette's `colorRole` tag —
  not the fixed `SAMPLED_W` array.
- The widest band in a strip is the preset's own DOMINANT-tier color(s), not unconditionally neutral.
- Two presets with different authored hierarchies visibly differ in strip PROPORTION, not just hue.
- Presets/sets with NO curated `story`/`colorRole` data (a user's own custom "Your Palettes" set) fall
  back to the current generic behavior — no regression there; this only changes CURATED preset cards.
- Non-regression: `p.key` (already a deliberately vivid peak-chroma color per palette, computed in
  `model.mjs` — the raw color data was never the problem) is still what each band renders.

## Scope / Open
- **In scope**: `presetTile()`'s strip construction only (`src/ui/app.js`).
- **Non-goal**: the hub-level category card's strip (`categoryCard()`/`c.strip` — one real dominant
  hex per volume, a different and already-more-reasonable mechanism) is NOT touched by this ticket. If
  it's found to have the same "looks generic" problem, that's a separate ticket.
- **Non-goal**: `buildTiles()`'s "Your Palettes" strip (saved user sets have no curated story to weight
  by — nothing to make more opinionated there).
- **Open (mechanical, confirm at build time)**: does `neutral` carry an explicit hierarchy weight in
  `story.groups`, or is it implicitly excluded from the dominant/supporting/accent split entirely (the
  sampled inspection during intake suggests the latter — neutral had no `colorRole` field on the
  palettes seen)? If excluded, neutral should get a small residual/fallback share, not zero and not 36%.
- **Open (design, confirm at build time)**: exact fallback behavior when a preset HAS `story.groups`
  but a palette is missing a `colorRole` tag (partial data) — reasonable default is "genuinely small
  fallback width," not re-adopting the old fixed template for that one swatch.

## Findings
- **2026-07-12** — Built and shipped as #269. `presetTile()` (`src/ui/app.js`) now weights each band
  by `story.groups[].pct` via `colorRole`: neutral fixed at 8%, dominant/supporting/accent scaled to
  fill the remaining 92% and split evenly across their palette count (dominant:1, supporting:3,
  accent:2), missing-`colorRole` gets a 5% fallback, no-`story.groups` falls back byte-for-byte to the
  original fixed template. Test coverage independently re-derives the expected math (not just an
  ordering check) plus a two-preset visible-difference assertion. Live-verified in real headless
  Chrome against the built artifact.
