# TKT-0011 — CI never diffs committed generated artifacts against a fresh regen (stale ui.html ships green)

## Summary
CI regenerates figma/plugin/ui.html + the asset files in the runner and tests THAT copy, but never compares it to what is committed — a contributor who skips the gen scripts ships stale UI to every Figma/npm/zip consumer on a green build. Also: CI uses `npm install` despite a committed lockfile. (docs/reference/reviews/2026-07-17-cto-platform.md CRITICAL-1 + MINOR-4.)

## Acceptance
- `git diff --exit-code` (scoped to generated artifacts) added after the build step in .github/workflows/ci.yml fails the run on any regen drift
- CI installs via `npm ci`
- A deliberately-stale artifact commit is demonstrated red before merge

## Links
- docs/reference/reviews/2026-07-17-cto-platform.md

## Findings

### 2026-07-17 — built; the gate bit TWICE before merge (#320)
Proof run 1 (by design): the PR's first commit carried a deliberately stale ui.html — CI went RED
on exactly the "Generated artifacts match committed" step. Proof run 2 (unplanned, the real
payoff): the regenerated commit STILL failed — the gate caught latent cross-version
non-determinism nobody knew about: `deriveNeutral`'s circular-mean hue was emitted as a raw
double, and V8's last-ulp trig variance between Node 22.14 (local) and CI's newer 22.x printed
`82.17507227608866` vs `82.1750722760886` in `src/ui/categories/architecture.js`. Fixed by
rounding all emitted oklch components to 4dp in gen-categories (`r4` — 1e-4 is far below
perceptual threshold), regenerating all 8 category files. `npm ci` also landed. Green on run 3.
